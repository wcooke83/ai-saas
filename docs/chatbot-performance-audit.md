# RAG Chatbot Performance Audit - 2026-03-21

## Executive Summary

**Total average request time: ~16,700ms. Target: under 4,000ms.**

The single biggest issue is a **critical embedding model mismatch bug** that causes 100% of similarity searches to return 0.00 confidence, triggering expensive live fetches on every single request. Fixing this one bug alone would eliminate ~7,500ms (the entire live fetch phase) from most requests, cutting total time roughly in half.

Beyond that bug, there are 12 additional optimization opportunities across parallelization, caching, database indexing, and threshold tuning.

## FINDING 1: CRITICAL BUG - Embedding Model Mismatch

**Severity: P0 | Estimated savings: ~7,500ms per request**

### Root Cause

The knowledge chunks (117 total) were embedded using **OpenAI `text-embedding-ada-002`** (1536 dimensions). However, `app_settings.embedding_model_id` is currently set to a **Gemini Embedding** model.

When a user query arrives, `resolveEmbeddingConfig()` in `src/lib/chatbots/knowledge/embeddings.ts` resolves to Gemini. The query embedding is generated by Gemini, but compared against OpenAI-generated chunk embeddings via cosine distance. These are **incompatible vector spaces** -- even with the same dimensionality (both 1536d), the vectors are semantically meaningless when cross-compared.

### Evidence

- 18 requests logged in last 30 days
- **ALL 18 show `rag_confidence = 0.000`**
- ALL 18 triggered live fetch
- Yet chunk-to-chunk similarity (same embedding model) shows scores of 0.88-0.90

### The Cascade

1. `match_knowledge_chunks` returns 0 similar chunks (threshold 0.7, all scores are ~0.0)
2. `bestSimilarity = 0` (`rag.ts` line 131-133)
3. `0 < LOW_CONFIDENCE_THRESHOLD (0.9)` is true (`rag.ts` line 142)
4. Live fetch triggers, adding ~7,500ms
5. 30 "priority chunks" still get returned (line 90-97) because they bypass similarity search, but they're noise -- unfiltered bulk chunks from the priority source

### Fix Options

- **Quick fix:** Change `app_settings.embedding_model_id` back to an OpenAI embedding model, or set it to NULL so auto-selection uses OpenAI (since `OPENAI_API_KEY` is present)
- **Proper fix:** Re-embed all chunks using the currently configured Gemini model. Add a safeguard that tracks which model was used per chunk and warns/blocks when the query model differs
- **Defensive fix:** Store `embedding_model` on each `knowledge_chunk` row. In `match_knowledge_chunks`, only compare chunks embedded with the same model as the query

## FINDING 2: `resolveEmbeddingConfig()` Called Twice Per Request

**Severity: P2 | Estimated savings: ~100-200ms**

**File:** `src/lib/chatbots/knowledge/embeddings.ts`

`getRAGContext()` calls:
1. `isEmbeddingsAvailable()` (`rag.ts` line 49) → calls `resolveEmbeddingConfig()`
2. `generateQueryEmbedding()` (line 65) → calls `generateEmbeddings()` → calls `resolveEmbeddingConfig()`

Each `resolveEmbeddingConfig()` call does:
- `getEmbeddingModel()` → `getAppSettings()` (cached 60s) → `getModelById()` (NO cache, hits DB every time)
- If that fails, `getActiveModelAndProvider()` (another DB query)

### Fix

Cache the resolved embedding config in a module-level variable with a TTL. Or resolve once at the start of `getRAGContext` and pass the config through.

## FINDING 3: Priority Source Design Flaw - 30 Unfiltered Chunks Always Included

**Severity: P1 | Estimated savings: ~500ms latency + significant token cost reduction**

**File:** `src/lib/chatbots/rag.ts` lines 90-97

When a knowledge source has `is_priority = true`, ALL its chunks (up to 30) are fetched and included in context regardless of relevance:

```typescript
supabase
  .from('knowledge_chunks')
  .select('id, content, metadata')
  .eq('chatbot_id', chatbot.id)
  .in('source_id', prioritySourceIds)
  .limit(30)
```

The single priority source has 115 chunks. 30 of these are blindly included in every response. Many are navigation link lists ("View All Tabletop Gaming", "View All Environmental Appliances") -- essentially website menu scraping, not useful knowledge.

### Problems

- Wastes tokens (30 x ~417 tokens = ~12,500 tokens of noise per request)
- Increases AI latency (more input tokens = slower inference)
- `bestSimilarity` is computed only from `similarChunks`, not from `priorityChunks`

### Fix

Priority sources should still use similarity search rather than brute-force inclusion. Or at minimum, run the priority chunks through the same similarity filter.

## FINDING 4: Live Fetch Has NO Caching

**Severity: P1 | Estimated savings: ~7,000ms on repeat queries**

**File:** `src/lib/chatbots/knowledge/live-fetch.ts`

`fetchPinnedUrlContent()` performs three expensive sequential operations with zero caching:

1. **HTML fetch + link extraction** (line 39-47): Fetches the pinned URL HTML, parses with cheerio, extracts all links. ~1-2s
2. **AI link picker** (line 67-100): Calls `generate()` to ask the AI which link answers the query. Full AI roundtrip ~2-3s
3. **Jina Reader fetch** (line 108-123): Fetches up to 2 URLs through `r.jina.ai/`. 30s timeout each. ~2-3s

None of these results are cached:
- The link list from step 1 changes rarely (website nav doesn't change between requests)
- The Jina Reader content for a given URL is static for minutes/hours
- Even the AI link-picking decision for similar queries could be cached

### Fix

Add multi-layer caching:
- Cache extracted links per pinned URL (TTL: 5-10 minutes)
- Cache Jina Reader content per URL (TTL: 10-30 minutes)
- Cache AI link-picking results by query hash + URL set (TTL: 5 minutes)
- Use in-memory Map or lightweight LRU cache (no Redis needed at this scale)

## FINDING 5: Live Fetch URL Timeout Too High (30s)

**Severity: P2 | Estimated savings: prevents 30s tail latency**

**File:** `src/lib/chatbots/knowledge/extractors/url.ts` line 10

```typescript
const FETCH_TIMEOUT = 30000; // 30 seconds
```

The Jina Reader and direct URL fetch both use a 30-second timeout. If a site is slow but not dead, the user waits up to 30 seconds.

### Fix

Reduce `FETCH_TIMEOUT` to 10-15s for live fetch scenarios. The 30s timeout is fine for async knowledge ingestion.

## FINDING 6: `match_knowledge_chunks` Lacks Composite Index

**Severity: P4 | Estimated savings: ~50-200ms at scale**

Current indexes:
- `idx_knowledge_chunks_embedding` -- HNSW on `(embedding vector_cosine_ops)` with DEFAULT params
- `idx_knowledge_chunks_chatbot` -- B-tree on `(chatbot_id)`

The query filters by `chatbot_id` first, then does vector similarity. PostgreSQL likely does a B-tree filter then sequential scan for vector similarity on the filtered set rather than using the HNSW index.

At 117 chunks this doesn't matter, but will become a bottleneck at scale.

### Fix

Create a partial HNSW index per chatbot, or restructure the query to use the HNSW index more effectively.

## FINDING 10: Widget Config Route Inline Agent Presence Check

**Severity: P4 | Architectural concern**

**File:** `src/app/api/widget/[chatbotId]/config/route.ts` lines 117-143

The widget config endpoint creates a second Supabase admin client and queries `agent_presence` inline. Despite `Cache-Control: public, max-age=60`, agent presence makes the response stale immediately.

### Fix

Return agent presence as a separate endpoint or use realtime subscription.

**Severity: P4 | Estimated savings: ~200ms on error paths**

**File:** `src/app/api/chat/[chatbotId]/route.ts` lines 871-896

The catch block re-queries the chatbot to get `user_id` for logging. The `chatbot` variable from the try block should be captured in a closure variable accessible from catch.

## FINDING 12: No Embedding Cache for Repeated Queries

**Severity: P2 | Estimated savings: ~1,400ms on repeat queries**

**File:** `src/lib/chatbots/knowledge/embeddings.ts`

Every query generates a fresh embedding via API call (~1,400ms). Common queries ("what are your hours?", "pricing", "return policy") would benefit from caching.

### Fix

Add an in-memory LRU cache for query embeddings keyed by normalized query text. TTL of 5-10 minutes. Memory footprint: ~720KB for all chunks -- trivially small.

---

## FINDING 13: `getMessages` Fetches with `SELECT *`

**Severity: P3 | Estimated savings: ~100ms**

**File:** `src/lib/chatbots/api.ts` lines 423-436

Messages table includes `context_chunks`, `metadata`, and `attachments` (all JSONB) that are never used by the chat route's history display.

### Fix

`select('id, role, content, created_at')` would significantly reduce payload size for conversations with attachments or large metadata.
